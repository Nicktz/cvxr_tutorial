Disciplined Convex Optimization in R

Chapter 12 Speed Considerations

12.1 Goals

  • Understand where CVXR spends a lot of time
  • Learn some tips for improving speed

CVXR allows one to formulate a convex problem at a high level, close to mathematical form, and solves it for you provided you follow the DCP rules. Behind the scenes, CVXR compiles the domain specific language and verifies the convexity of the problem before dispatching it to a solver. Therefore, it is generally slower than tailor-made solutions for a given problem.

12.2 Example Problem

To understand the speed issues, let us consider the global warming data from the Carbon Dioxide Information Analysis Center (CDIAC) again. The data points are the annual temperature anomalies relative to the 1961–1990 mean. We will fit the nearly-isotonic approximation \(\beta \in {\mathbf R}^m\) by solving

\[ \begin{array}{ll} \underset{\beta}{\mbox{Minimize}} & \frac{1}{2}\sum_{i=1}^m (y_i - \beta_i)^2 + \lambda \sum_{i=1}^{m-1}(\beta_i - \beta_{i+1})_+, \end{array} \] where \(\lambda \geq 0\) is a penalty parameter and \(x_+ =\max(x,0)\).

This can be solved as follows.

data(cdiac)
y <- cdiac$annual
m <- length(y)
lambda <- 0.44
beta <- Variable(m)
obj <- 0.5 * sum((y - beta)^2) + lambda * sum(pos(diff(beta)))
prob <- Problem(Minimize(obj))
soln <- solve(prob)
betaHat <- soln$getValue(beta)

This is the recommended way to solve a problem.

However, suppose we wish to construct bootstrap confidence intervals for the estimate using 100 resamples. It is clear that this computation time can quickly become limiting.

Below, we show how one can get at the problem data and directly call a solver to get faster results.

12.3 Profile the code

Profiling a single fit to the model is useful to figure out where most of the time is spent.

data(cdiac)
y <- cdiac$annual
profvis({
    beta <- Variable(m)
    obj <- Minimize(0.5 * sum((y - beta)^2) + lambda * sum(pos(diff(beta))))
    prob <- Problem(obj)
    soln <- solve(prob)
    betaHat <- soln$getValue(beta)
})

It is especially instructive to click on the data tab and open up the tree for solve to see the sequence of calls and cumulative time used.

The profile shows that most of the total time (2400ms for one of our runs) is spent in the call to is_dcp generic (about 2000ms). This generic is responsible for ensuring that the problem is DCP-compliant by checking the nature of each of its components. The actual solving took a much smaller fraction of the time.

12.4 Directly Calling the Solver

We are mathematically certain that our problem is convex, so we can avoid the is_dcp hit. We can obtain the the problem data for a particular solver (like ECOS or SCS) using the function get_problem_data and directly hand that data to the solver to get the solution.

prob_data <- get_problem_data(prob, solver = "ECOS")

ASIDE: How did we know ECOS was the solver to use? Future versions will provode a function to match a solver to a problem. (Actually, it is available already, but not exported yet!). For now, a single call to solve with the verbose option set to TRUE can provide that information.

soln <- solve(prob, verbose = TRUE)

Now that we have the problem data and know which solver to use, we can call the ECOS solver with the right arguments. (The ECOS solver is provided by the package ECOSolveR which CVXR imports.)

solver_output <- ECOSolveR::ECOS_csolve(c = prob_data[["c"]],
                                        G = prob_data[["G"]],
                                        h = prob_data[["h"]],
                                        dims = prob_data[["dims"]],
                                        A = prob_data[["A"]],
                                        b = prob_data[["b"]])

Finally, we can obtain the results by asking CVXR to unpack the solver results for us. (See ?unpack_results for further examples.)

direct_soln <- unpack_results(prob, "ECOS", solver_output)

12.5 Profile the Direct Call

We can profile this direct call now.

profvis({
    beta <- Variable(m)
    obj <- Minimize(0.5 * sum((y - beta)^2) + lambda * sum(pos(diff(beta))))
    prob <- Problem(obj)
    prob_data <- get_problem_data(prob, solver = "ECOS")
    solver_output <- ECOSolveR::ECOS_csolve(c = prob_data[["c"]],
                                            G = prob_data[["G"]],
                                            h = prob_data[["h"]],
                                            dims = prob_data[["dims"]],
                                            A = prob_data[["A"]],
                                            b = prob_data[["b"]])
    direct_soln <- unpack_results(prob, "ECOS", solver_output)
})

For one of our runs, the total time went down from \(2400ms\) to \(690ms\), more than a 3-fold speedup! In cases where the objective function and constraints are more complex, the speedup can be more than 10-fold.

12.6 Same Answer?

Of course, we should also verify that the results obtained in both cases are same.

identical(betaHat, direct_soln$getValue(beta))
## [1] TRUE
